# Lore Agent vs Karpathy's LLM Wiki — Comparative Analysis

**Purpose**: This document is written for external AI review. It compares the lore-agent project with Karpathy's LLM Wiki gist, identifies gaps, and proposes improvements.

**Date**: 2026-04-10

---

## 1. Systems Overview

### Lore Agent

A local knowledge retrieval and research agent with an MCP (Model Context Protocol) server interface. It allows AI coding assistants (Claude Code, Copilot) to query, save, and browse domain knowledge cards through structured tools.

**Architecture**:
- MCP server (`mcp_server.py`) exposing 3 tools: `query_knowledge`, `save_research`, `list_knowledge`
- BM25 + embedding hybrid search pipeline for retrieval
- Knowledge cards: markdown files with YAML frontmatter, organized by topic domain
- Research pipeline: web research → synthesize → distill → promote → reindex
- Lifecycle states: draft → reviewed → trusted → stale → deprecated
- 2 card types: **knowledge** (concepts, facts) and **method** (procedures, implementations)

**Key design decisions**:
- Cards are generated by AI during research, not manually authored
- Schema-validated structured answers (JSON Schema) are distilled into markdown cards
- Index is rebuilt after each card insertion for immediate retrievability
- Method cards include Expected Output and Concrete Example sections so AI can implement directly

### Karpathy's LLM Wiki

A personal wiki system where an LLM acts as a research assistant. The user asks questions, the LLM researches and writes wiki pages, and those pages persist across sessions.

**Architecture** (from the gist):
- Single persistent wiki file (`wiki.md`) or multi-file setup
- Manual prompt-driven workflow: user asks, LLM writes, file persists
- Cross-references via `[[wiki-links]]`
- Lint checks for content health (orphans, contradictions, staleness)
- Changelog in `log.md` tracking additions and changes
- Obsidian-compatible markdown for rich navigation

**Key design decisions**:
- Human-initiated, human-curated — LLM is the writer, human is the editor
- Wiki-links create a graph structure between topics
- Lint enforces knowledge base hygiene
- Simple file-based persistence, no server or protocol layer

## 2. Similarities

| Aspect | Lore Agent | LLM Wiki |
|--------|-----------|----------|
| Persistence | Markdown files with frontmatter | Markdown files |
| AI-driven content | Research → synthesis → card | Research → wiki page |
| Retrieval | BM25 + embedding hybrid search | File reading / Obsidian search |
| Structure | Schema-validated JSON → markdown | Freeform markdown |
| Goal | AI-assisted domain knowledge | Personal knowledge wiki |

Both systems solve the same fundamental problem: **LLM context is ephemeral, but knowledge should persist**. Both use markdown files as the persistence layer and rely on AI to generate structured content.

## 3. Differences

### Where Lore Agent is Stronger

1. **Structured schema validation**: Every answer passes through `answer.schema.json` validation before becoming a card. This enforces consistency (required fields, confidence levels, evidence references). LLM Wiki has no schema enforcement.

2. **Programmatic tool interface**: MCP protocol lets any AI assistant query and save knowledge without manual prompting. LLM Wiki requires the user to explicitly say "write this to the wiki."

3. **Hybrid search pipeline**: BM25 + embedding retrieval with scoring gives ranked results. LLM Wiki relies on the LLM reading the entire file or the user browsing in Obsidian.

4. **Lifecycle governance**: Card states (draft → reviewed → trusted → stale → deprecated) with transition validation and governance CLI. LLM Wiki has no formal lifecycle.

5. **Automatic domain classification**: Cards are auto-classified into domains and types based on query patterns. LLM Wiki depends on the author to categorize.

6. **Evidence traceability**: Each claim links to evidence IDs and source URLs. LLM Wiki doesn't enforce evidence chains.

### Where LLM Wiki is Stronger

1. **Card interconnection**: `[[wiki-links]]` create a navigable graph between topics. Lore Agent's "Related Concepts" section is plain text — no bidirectional links, no graph traversal.

2. **Content lint**: Built-in checks for orphans (unlinked pages), contradictions (conflicting claims across pages), and staleness (outdated information). Lore Agent has schema validation but no content-level health checks.

3. **Changelog**: `log.md` tracks what was added, changed, or deprecated with timestamps. Lore Agent has no audit trail beyond git history.

4. **Answer-writeback**: Good answers to user questions automatically get filed as new wiki pages. In Lore Agent, knowledge only enters the system through the explicit `save_research` tool call — natural Q&A that produces useful knowledge is lost.

## 4. Identified Gaps and Proposed Improvements

### Gap 1: Cards are isolated — no graph structure

**Current state**: "Related Concepts" is a text list of card IDs. No bidirectional links, no way to traverse the knowledge graph.

**Proposal**: Add `[[card-id]]` wiki-link syntax to knowledge cards. During indexing, extract links and build a bidirectional reference map. Add a `related_cards` tool that returns cards linked to a given card.

**Implementation notes**:
- Parse `[[...]]` links during index build
- Store backlinks in index metadata
- Add `--backlinks` flag to `list_knowledge` or new `graph` tool

### Gap 2: No content lint

**Current state**: Schema validation checks field presence and types but not content quality. No detection of orphan cards, contradictions, or stale information.

**Proposal**: Add a `lint` tool/command that checks:
- **Orphan detection**: Cards with no incoming or outgoing links
- **Contradiction detection**: Cards on the same topic with conflicting claims
- **Staleness detection**: Cards with `updated_at` older than a configurable threshold
- **Schema drift**: Cards whose frontmatter doesn't match current schema expectations

**Implementation notes**:
- Extend `knowledge_governance.py` with lint subcommand
- Orphan detection requires graph structure (depends on Gap 1)
- Contradiction detection can use embedding similarity + claim comparison

### Gap 3: No changelog

**Current state**: No structured log of knowledge base changes. Git history exists but isn't queryable through MCP tools.

**Proposal**: Maintain a `knowledge/changelog.md` file that records:
- Card created (with topic, type, query)
- Card updated (what changed)
- Card deprecated (reason)
- Lifecycle transitions

**Implementation notes**:
- Append to changelog in `build_knowledge_card()` and `transition_card()`
- Add `changelog` tool or `--since` filter to `list_knowledge`

### Gap 4: No answer-writeback

**Current state**: Useful knowledge produced during conversations is lost unless the user explicitly calls `save_research`. Many good answers in natural Q&A never become cards.

**Proposal**: Add a `capture_answer` tool that extracts a structured answer from conversation context and files it as a draft card. AI assistants should be prompted to call this when they produce a substantive answer that isn't already in the knowledge base.

**Implementation notes**:
- Simpler than `save_research` — no research evidence required
- Lower confidence level (draft, unverified)
- Could be triggered by AI self-assessment ("this answer is novel and useful")
